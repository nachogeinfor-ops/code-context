## Why these four tracks (and this order)

Sprint 8's eval suite gave us hard evidence about where v1.0.0 is weak:

| Pain | Evidence | Track |
|---|---|---|
| Embedding model is general-purpose, not code-trained | Vector NDCG@10 0.44 on a C# repo (MiniLM hasn't seen C# in pre-training) | A — Quality |
| Hybrid AND-of-tokens degrades to vector-only on long natural-language queries | Hybrid 0.4172 ≈ vector_only 0.4384, 3/35 BM25 queries returned [] | A — Quality |
| `find_references` ranks docs/archive over live source code | Sprint 5 smoke — top-5 refs for `ExecuteAsync` were all from `docs/archive/*.md` | A — Quality |
| Tree-sitter only covers Py/JS/TS/Go/Rust; C#/Java/C++ fall back to line chunker | WinServiceScheduler has 1169 .cs files; all chunked by line-window | B — Reach |
| Reranker p50 = 6.3 s on CPU, p95 = 13 s | Sprint 8 eval — unusable as default; explains why `CC_RERANK=off` is the default | C — Latency |
| Eval set is 35 queries, single repo, single language | Hard to claim "improved" without more coverage | D — Eval expansion |

**Order rationale: D → A → B → C.** Without an expanded eval (D), we can't honestly measure whether A/B/C are improvements. D is also the cheapest sprint, so getting it out of the way first reduces risk on every subsequent sprint.

## Hard constraints (apply to every v1.x sprint)

### v1 stability commitment

The public API surface in [`../v1-api.md`](../v1-api.md) is frozen for v1.x. **Adding** new MCP tools, env vars, CLI subcommands, or Python imports is a minor bump (v1.X.0). **Removing or renaming** anything listed there requires v2.

In practice for these sprints:

- Sprint 10 (Quality) cannot rename `CC_EMBEDDINGS_MODEL` or change its default in a way that drops `all-MiniLM-L6-v2` — that would break existing users' caches via the model-id staleness check (which is intentional, but disruptive). Default change is OK if accompanied by clear CHANGELOG warning + auto full-reindex on first v1.2.0 startup (already the existing `dirty_set` behavior).
- Sprint 11 (Reach) cannot remove existing tree-sitter grammars; only add.
- Sprint 12 (Latency) can change the `CC_RERANK_MODEL` default (it's optional anyway), but cannot remove `CC_RERANK` env var.
- Sprint 9 (Eval) is mostly internal — adds files under `benchmarks/eval/`, no public-API impact.

### PyPI release pipeline

Already configured. Each `v*` tag push triggers `.github/workflows/release.yml`:

1. Build job: `python -m build`, `twine check`.
2. Publish job: OIDC handshake with PyPI Trusted Publisher (project `code-context-mcp`, owner `nachogeinfor-ops`, repo `code-context`, env `release`). No secrets.

**Don't push the tag without user authorization.** The user explicitly gates each release. The pattern from Sprints 6-8: tag locally → user says "push" → push happens → PyPI updates within ~1 min → smoke in clean venv.

### Eval gating (post-Sprint 9)

After Sprint 9 lands, every subsequent sprint MUST run the expanded eval suite as part of its acceptance criteria:

- NDCG@10 (hybrid_rerank config) must not regress more than 0.02 absolute vs the prior tag's baseline.
- p50 latency (hybrid_rerank) must not regress more than 50% vs the prior tag's baseline.
- New configs / models added by the sprint produce their own row in `benchmarks/eval/results/v1.X.0_*.csv`.

The eval is opt-in for CI (it's CPU-bound and slow); the decision rule is "ran by the maintainer before tag push, recorded in the sprint's benchmark doc."

## Per-sprint summaries

### Sprint 9 — Eval coverage (v1.1.0)

**Goal:** Make the eval suite a real regression net. Three deliverables:

1. **Expanded query set**: from 35 → ~120 queries. Buckets:
   - 60 queries against WinServiceScheduler (C#) — current set + 25 new.
   - 30 queries against the `tiny_repo` fixture or a real Python repo (e.g. `requests` or `httpx` snapshot).
   - 30 queries against a TypeScript repo (e.g. snapshot of `mcp` SDK or similar).

2. **Multi-repo runner**: `runner.py` accepts a config that lists multiple `(repo, queries)` pairs and emits a combined CSV plus per-repo breakdown. Lets us see if quality is consistent across languages or only good on the training distribution.

3. **CI drift gate**: a new optional CI workflow (`.github/workflows/eval.yml`) that runs the eval against `tiny_repo` on every PR and posts NDCG@10 delta vs `main`. Optional because it adds CI time; gated on a label `run-eval`.

Deliverable: `benchmarks/eval/v1.1.0_*.csv` baseline with all 3 repos, README updated. Plus a `benchmarks/eval/queries/{csharp,python,typescript}.json` split.

**Risks:**
- Hand-curating 90 new queries is the long pole. Bias toward queries we'd actually ask Claude Code in those repos.
- TS / Python repos need to be checked into the test fixtures or downloaded via a script (don't bloat the repo).

**Acceptance:** ≥120 queries across 3 repos; runner emits combined + per-repo CSVs; CI eval workflow green on a manual dispatch.

### Sprint 10 — Quality (v1.2.0)

**Goal:** Move NDCG@10 (hybrid_rerank) from 0.46 → 0.55+ on the expanded eval.

Three threads:

1. **Code-trained embeddings**: replace default model from `all-MiniLM-L6-v2` (general-purpose, 384-dim) to `BAAI/bge-code-v1.5` (code-trained, 1024-dim) for the local provider. Existing `MODEL_REGISTRY` makes this a one-line swap. Larger model → slower indexing; profile and document.

2. **Stop-word filter on BM25 query**: extend `_sanitise` to drop common English stop words ("how", "is", "the", "what", "where", "do", "we", "a") before AND-ing tokens. So "how is settings.json loaded" → ["settings", "json", "loaded"] → AND of 3 content tokens → matches docs that have all 3. Tunable list; start with NLTK's English set.

3. **`find_references` source-priority ranking**: post-filter SymbolRef results so paths under the repo's primary source dirs (auto-detected: `src/`, `lib/`, `app/`, or whatever has the most indexed files) rank above paths under `docs/`, `tests/`, `examples/`, `archive/`. Stable sort: source-first, then by line number. Adds one extra column to the symbol DB schema or a runtime classification.

**Risks:**
- bge-code-v1.5 is much larger (1.6 GB vs 90 MB MiniLM). The "default install pulls the model on first run" promise becomes painful on slow connections. Mitigation: keep MiniLM as a fast fallback via `CC_EMBEDDINGS_MODEL=fast`; bump default but document the size.
- Stop-word filter may over-strip ("set" → preserved? "by" → dropped?). Eval will tell us.
- find_references priority sort is heuristic; document it as "best-effort, configurable via `CC_SYMBOL_RANK=source-first|natural`".

**Acceptance:** NDCG@10 (hybrid_rerank) ≥ 0.55 on the v1.1.0 eval set; no regression on hit@10; documented model-size trade-off.

### Sprint 11 — Reach (v1.3.0)

**Goal:** Tree-sitter chunking for the languages that today fall through to the line chunker. Drop the v0.2 5-language list to add C#, Java, C++, Markdown.

- C# (was in the docs as supported but actually only line-chunked because no parser was wired in v0.2; verify in `chunker_treesitter.py`).
- Java (huge corpus, many users).
- C++ (`.cpp`, `.hpp`, `.h`).
- Markdown (chunk by heading hierarchy — different node names but same visit pattern).

For each: (a) load the tree-sitter grammar (already vendored via `tree-sitter-language-pack`); (b) define the node-type → chunk-kind mapping (`class_declaration` → "class", `method_declaration` → "method", etc.); (c) add `extract_definitions` for the symbol index.

**Risks:**
- C++ is template-heavy; tree-sitter parses templates but our chunker may emit huge chunks. Cap by lines + AST depth.
- Markdown chunks need different content-hash semantics (whitespace variations); document.

**Acceptance:** All 4 new languages have unit tests + integration tests; the smoke against WinServiceScheduler shows .cs files now chunked at function-boundary granularity (chunks_per_file goes from ~7 line-windows to ~3 AST chunks per typical file); eval doesn't regress.

### Sprint 12 — Latency (v1.4.0)

**Goal:** Make `CC_RERANK=on` viable as a default. Today p50 = 6.3 s on CPU.

- **Distilled cross-encoder default**: replace `cross-encoder/ms-marco-MiniLM-L-6-v2` (~22 M params) with `cross-encoder/ms-marco-MiniLM-L-2-v2` (~4 M params) or a quantized INT8 variant. Measure quality drop vs latency win.
- **GPU auto-detect**: if `torch.cuda.is_available()`, run rerank on CUDA. Ship as a config-free default.
- **Embed-result cache**: hash the query text → cache the embedding for the session. Dedup repeat queries (Claude Code re-asks the same thing across turns).
- **Batched rerank**: today the cross-encoder runs 15 forward passes per query sequentially. Batch them (`predict([(q, c) for c in candidates], batch_size=8)`).

**Risks:**
- Distilled rerank may drop NDCG@10 by 0.02-0.05; trade-off explicit in CHANGELOG.
- GPU detection on Windows with stock torch is shaky; smoke on the actual user's machine before defaulting.

**Acceptance:** Reranker p50 ≤ 1.5 s on CPU (4× speedup) AND ≤ 100 ms on CUDA; NDCG@10 drop ≤ 0.03 vs current cross-encoder.

## Out-of-scope for v1.x

These were on the original roadmap but stay deferred:

- ColBERT-style late-interaction retrieval (architectural rework)
- Multi-repo cross-search (changes the cache layout)
- Cloud-hosted shared-index for teams (introduces a server component)
- Native non-Python packaging
- Telemetry / opt-in usage metrics
- Symbol-graph extraction (call graphs, type hierarchies)

Each of these merits its own brainstorm + spec when the time comes.
